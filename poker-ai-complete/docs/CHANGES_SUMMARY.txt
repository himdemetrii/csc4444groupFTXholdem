1. Legal Action Detection
   ─────────────────────
   BEFORE: Guessed CHECK/CALL availability
   AFTER:  Follows engine rules strictly
   
   Engine rules:
   • CHECK only when toCall == 0
   • CALL only when toCall > 0
   • Can't CHECK when there's a bet → must CALL or FOLD

2. Action Validation
   ─────────────────────────
   BEFORE: Converted illegal CHECK to FOLD
   AFTER:  Smart conversion CHECK ↔ CALL based on game state
   
   Examples:
   • Model says CHECK but bet exists → converts to CALL
   • Model says CALL but no bet → converts to CHECK
   • Prevents server rejecting actions

3. Raise Amount Validation
   ───────────────────────
   BEFORE: Fixed BB multipliers, no chip validation
   AFTER:  Checks available chips, handles all-in
   
   Now handles:
   • Short stack all-in calls
   • Short stack all-in raises
   • Proper raise sizing

• Using heuristics as workaround

For now, the client works with smart fallbacks and will adapt to whatever
the server accepts.
